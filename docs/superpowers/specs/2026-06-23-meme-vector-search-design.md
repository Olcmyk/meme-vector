# 中文表情包向量搜索工具设计文档

## 项目概述

构建一个基于语义搜索的中文表情包搜索网站，用户可以通过自然语言描述（如"一只开心的猫"）搜索到相关的表情包图片。

**目标用户：** 中等流量的公开网站（数百到数千用户）

**核心功能：** 语义搜索，无需用户账户系统

## 技术栈

- **前端框架：** Next.js 14+ (App Router)
- **向量数据库：** Pinecone
- **向量嵌入模型：** Qwen3-VL-Embedding-8B（通过硅基流动 API）
- **部署平台：** Vercel
- **数据源：** emo-visual-data (https://github.com/LLM-Red-Team/emo-visual-data)

## 系统架构

### 整体架构

系统分为两个独立的部分：

1. **数据准备流程**（一次性离线处理）
   - 读取 emo-visual-data 数据集
   - 使用 Qwen3-VL-Embedding-8B 生成向量嵌入
   - 批量导入 Pinecone 向量数据库

2. **Web 应用**（Next.js）
   - 前端：搜索界面和结果展示
   - 后端 API：处理搜索请求和向量检索
   - 部署在 Vercel

### 架构图

```
用户浏览器
    ↓
Next.js 前端 (Vercel)
    ↓
Next.js API Routes
    ↓
    ├─→ 硅基流动 API (生成查询向量)
    └─→ Pinecone (向量检索)
```

## 核心组件

### 1. 数据处理脚本

**位置：** `/scripts/prepare-data.ts` 或 `/scripts/prepare-data.py`

**功能：**
- 从 emo-visual-data 仓库读取表情包数据
- 解析 LLM 标注的描述文本
- 批量调用硅基流动 API 生成向量嵌入
- 将向量和元数据上传到 Pinecone
- 支持断点续传和错误重试

**关键实现：**
- 批处理以提高效率（如每批 50-100 个）
- 添加进度条显示处理状态
- 记录失败条目到日志文件
- 避免重复处理已存在的数据

### 2. Next.js 应用结构

```
/app
  /page.tsx                 # 搜索主页
  /api
    /search
      /route.ts            # 搜索 API 端点
/components
  /SearchBar.tsx           # 搜索框组件
  /MemeGrid.tsx            # 表情包网格展示
  /MemeCard.tsx            # 单个表情包卡片
  /LoadingState.tsx        # 加载状态
/lib
  /pinecone.ts             # Pinecone 客户端封装
  /embedding.ts            # 硅基流动 API 调用
  /types.ts                # TypeScript 类型定义
```

### 3. API 端点设计

**搜索端点：** `GET /api/search`

**请求参数：**
- `q` (string, required): 搜索查询文本
- `limit` (number, optional): 返回结果数量，默认 20

**响应格式：**
```json
{
  "success": true,
  "results": [
    {
      "id": "meme_0001",
      "imageUrl": "https://...",
      "description": "一只开心的橘猫",
      "score": 0.85
    }
  ],
  "total": 20
}
```

**错误响应：**
```json
{
  "success": false,
  "error": "搜索服务暂时不可用，请稍后重试"
}
```

## 数据模型

### Pinecone 索引配置

**索引名称：** `meme-search`

**向量维度：** 根据 Qwen3-VL-Embedding-8B 的输出维度（需要在实现时确认，通常是 768 或 1024）

**距离度量：** Cosine Similarity

### 向量记录结构

每个向量包含：

```typescript
{
  id: string,              // 唯一标识符，如 "meme_0001"
  values: number[],        // 向量嵌入数组
  metadata: {
    imageUrl: string,      // 表情包图片 URL
    description: string,   // LLM 标注的描述
    tags?: string[],       // 可选的标签数组
    source: string         // 数据来源 "emo-visual-data"
  }
}
```

### 图片存储方案

**选项 1：** 使用 emo-visual-data 仓库的原始 URL（GitHub raw content）
**选项 2：** 下载到 Vercel Blob Storage（如果需要更好的性能和控制）
**选项 3：** 托管到 CDN（如 Cloudflare R2）

**推荐：** 先使用选项 1 快速启动，如果性能成为瓶颈再迁移到选项 2 或 3。

## 搜索流程

### 完整流程

1. **用户输入**
   - 用户在搜索框输入"一只开心的猫"
   - 前端进行输入验证（非空，长度限制）

2. **前端处理**
   - 应用 debounce（500ms）避免频繁请求
   - 显示加载状态
   - 发送 GET 请求到 `/api/search?q=一只开心的猫&limit=20`

3. **后端处理**
   - 接收查询文本
   - 调用硅基流动 API 生成查询向量
   - 使用向量在 Pinecone 中搜索相似内容
   - 获取 top 20 相似结果
   - 格式化并返回响应

4. **结果展示**
   - 渲染表情包网格（响应式布局）
   - 显示图片和描述文本
   - 支持点击复制图片或下载

### 性能优化

- 使用 Next.js Image 组件优化图片加载
- API 响应启用压缩
- 合理的 Pinecone `top_k` 值（20-30）
- 前端添加防抖和请求取消

## 错误处理

### API 调用失败

**硅基流动 API 失败：**
- 实现指数退避重试（最多 3 次）
- 返回用户友好的错误信息
- 记录错误日志供排查

**Pinecone 查询失败：**
- 捕获异常并返回 500 错误
- 显示降级的错误页面
- 记录详细错误信息

### 搜索边界情况

| 情况 | 处理方式 |
|------|---------|
| 空查询 | 前端验证，提示"请输入搜索内容" |
| 查询过长（>100 字符） | 截断或提示用户简化 |
| 无搜索结果 | 显示"未找到相关表情包，试试其他关键词" |
| 网络延迟 | 显示加载骨架屏，超时提示 |

### 数据处理容错

- 部分表情包处理失败时跳过，继续处理其他
- 记录失败条目到 `failed.log`
- 支持增量导入，避免重复处理
- 提供手动重试失败条目的功能

### 速率限制

**前端：**
- 搜索输入 debounce 500ms
- 取消未完成的请求（当用户继续输入时）

**后端（可选）：**
- 简单的 IP 限流（如每分钟 30 次请求）
- 防止滥用和成本失控

## 技术决策记录

### 为什么选择 Pinecone？

- 托管服务，无需运维
- 支持高效的向量相似度搜索
- 有免费额度适合项目启动
- API 简单易用

### 为什么选择 Next.js？

- 同时支持前后端开发
- 与 Vercel 深度集成
- App Router 提供现代化的开发体验
- 内置图片优化和性能优化

### 为什么选择方案 A（纯在线搜索）？

- 架构简单，易于维护
- 搜索质量最高（每次实时生成向量）
- 成本可控（硅基流动价格合理）
- 可扩展性好（后续可加缓存）

## 风险和缓解

### 成本风险

**风险：** 如果流量暴增，API 调用成本可能增加

**缓解：**
- 监控 API 使用量
- 设置成本告警
- 必要时引入缓存层
- 实施速率限制

### 依赖风险

**风险：** 依赖第三方服务（硅基流动、Pinecone）的可用性

**缓解：**
- 实现重试机制
- 友好的错误提示
- 监控服务状态
- 关键时刻可切换到备用方案

### 数据质量风险

**风险：** emo-visual-data 的标注可能不完全准确

**缓解：**
- 测试阶段验证搜索质量
- 收集用户反馈
- 必要时补充或修正标注
- 考虑混合多个数据源
